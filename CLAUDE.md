# 项目协作规范

本项目技术栈：

- JDK 17
- Spring Boot 3
- Maven
- MyBatis-Plus
- Redis
- MySQL 8

Claude 在本项目中工作时，必须遵守以下规则。

---

## 1. 基础工作原则

- 优先保证正确性、可维护性、可测试性，不为了“炫技”引入复杂实现。
- 优先做最小改动，避免无关重构。
- 修改前先理解上下文，再动手改代码。
- 修改后必须说明：
    - 改了哪些文件
    - 改了什么行为
    - 是否影响数据库、缓存、接口、事务
    - 跑了哪些验证
    - 还存在哪些风险

---

## 2. 构建与运行规范

- 本项目统一使用 JDK 17。
- 本项目统一使用 Maven Wrapper：
    - `./mvnw`
- 除非明确要求，不要使用系统全局 `mvn`。
- 修改前先查看：
    - `pom.xml`
    - `src/main/resources/application.yml`
    - `src/main/resources/application-*.yml`
- 常用命令：
    - 编译：`./mvnw clean compile`
    - 测试：`./mvnw test`
    - 打包：`./mvnw clean package`
    - 启动：`./mvnw spring-boot:run`
- 优先执行最小必要验证，不要默认跑全量耗时流程。

---

## 3. 代码风格规范

- 使用清晰、稳定、可读的命名。
- 不使用无意义缩写，除非是业界通用缩写。
- 禁止使用通配符导入：
    - 不允许 `import xxx.*`
- 优先保持方法短小、职责单一。
- 优先使用 `final` 修饰不会变化的局部变量和参数。
- 不为了抽象而抽象，不引入没有明确收益的设计模式。
- 非必要不要引入反射、动态代理、复杂泛型技巧。
- 对公共接口、复杂逻辑、边界处理添加必要注释，但不要写无信息量注释。

---

## 4. Spring Boot 3 规范

- Controller 只负责：
    - 接收参数
    - 调用 Service
    - 返回统一响应
- 不在 Controller 中写业务逻辑。
- Service 层负责核心业务编排。
- Repository / Mapper 层只负责数据访问。
- 优先使用构造器注入，禁止字段注入：
    - 不要使用 `@Autowired` 在字段上注入
- 事务边界放在 Service 层，而不是 Controller 层。
- 异常处理尽量统一收敛到全局异常处理器。
- 新增接口时，优先遵循已有 REST 风格、返回结构、错误码风格。
- 涉及配置项时，优先使用 `@ConfigurationProperties`，不要散落硬编码。

---

## 5. DTO / VO / Entity 规范

- 不允许在接口层直接暴露数据库实体类。
- 请求对象使用 Request DTO。
- 响应对象使用 Response DTO / VO。
- Entity 只用于持久化层。
- DTO、VO、Entity 之间的转换应保持明确，不要混用。
- 字段命名必须体现业务语义，不要把数据库命名习惯直接泄漏到接口层。

---

## 6. MyBatis-Plus 规范

- 优先使用 MyBatis-Plus 提供的标准能力实现单表 CRUD。
- 使用 `LambdaQueryWrapper` / `LambdaUpdateWrapper` 时，优先保证可读性。
- 条件构造必须明确，避免隐式条件导致误查询或误更新。
- 批量更新、批量删除必须显式评估影响范围。
- 涉及复杂 SQL 时：
    - 优先保证正确性和可读性
    - 明确索引使用风险
    - 必要时保留 SQL 说明
- 不要滥用通用查询，避免写出不可维护的“万能方法”。
- 分页查询必须走统一分页方式，不要手写重复分页逻辑。
- 修改查询逻辑时，要主动检查是否存在 N+1 问题。

---

## 7. MySQL 8 规范

- 所有表结构变更必须通过数据库变更脚本管理，不允许口头约定改库。
- 涉及表结构调整时，必须说明：
    - 是否影响已有数据
    - 是否需要补默认值
    - 是否需要索引调整
    - 是否存在锁表风险
- 查询优化优先关注：
    - 索引命中
    - where 条件可用性
    - 排序与分页代价
    - 回表风险
- 禁止拼接 SQL 处理外部输入。
- 删除、更新操作必须明确 where 条件，避免误伤全表。
- 涉及唯一约束、幂等、防重复提交时，要同时考虑数据库层兜底能力。

---

## 8. Redis 规范

- Redis 用于缓存、分布式锁、热点数据或短期状态存储时，必须明确用途。
- 新增缓存时必须说明：
    - key 设计
    - TTL
    - 缓存值结构
    - 更新策略
    - 一致性风险
- key 命名建议：
    - `业务域:对象类型:主键[:附加维度]`
- 不允许使用含义不清的 key 名。
- 读取缓存时要考虑：
    - 缓存穿透
    - 缓存击穿
    - 缓存雪崩
- 修改数据库逻辑时，必须同步评估缓存失效策略。
- 使用分布式锁时：
    - 明确锁粒度
    - 明确超时时间
    - 明确失败重试策略
    - 避免死锁和误释放
- 不要把 Redis 当数据库长期存储核心业务数据，除非系统已有明确设计。

---

## 9. 接口设计规范

- 保持 REST 风格一致性。
- URL 命名使用名词，不要使用动词堆砌。
- GET 用于查询，POST 用于新增，PUT/PATCH 用于更新，DELETE 用于删除。
- 分页接口统一返回分页结构。
- 返回值遵循项目统一响应体规范。
- 错误信息对外应清晰、稳定，不暴露内部实现细节。
- 参数校验优先使用 Bean Validation：
    - `@NotNull`
    - `@NotBlank`
    - `@Size`
    - `@Min`
    - `@Max`
    - 等
- 任何来自外部的输入都视为不可信。

---

## 10. 安全规范

- 禁止在日志中输出：
    - 密码
    - token
    - access key
    - secret
    - 完整手机号、身份证号、银行卡号等敏感信息
- 涉及权限校验时，必须明确检查：
    - 是否登录
    - 是否有角色权限
    - 是否有数据权限
    - 是否存在越权访问
- 文件上传、文件读取必须检查：
    - 类型
    - 大小
    - 路径穿越风险
- 外部 URL 调用必须评估 SSRF 风险。
- 涉及登录、鉴权、会话、令牌刷新时，优先保持现有安全机制一致。
- 不要在代码中硬编码密钥、密码、地址、证书内容。

---

## 11. 事务规范

- 事务应放在 Service 层。
- 不要把大段无关逻辑包进同一个事务。
- 涉及“数据库 + 缓存 + 消息”的场景时，必须明确一致性策略。
- 需要幂等保证的操作，必须显式说明幂等方案。
- 对外部系统调用不要轻易包进长事务。
- 遇到批量处理时，要评估事务范围是否过大。

---

## 12. 日志规范

- 日志要能帮助定位问题，但不能泄露敏感信息。
- 错误日志应包含：
    - 业务上下文
    - 关键标识
    - 异常信息
- 不要滥用 info 打印大对象。
- 避免在循环中大量打印日志。
- 对关键链路应保证日志可追踪，但不重复刷屏。

---

## 13. 测试规范

- 非微小改动必须补测试或更新测试。
- 修 bug 时，优先增加一个可以复现问题的测试。
- 纯业务逻辑优先写单元测试。
- 涉及数据库、Spring 容器、接口调用的改动，优先补集成测试。
- 不做无意义 mock。
- 测试命名要体现业务场景，不要只写 shouldTest1 这类名字。
- 至少运行与改动直接相关的测试。

---

## 14. 变更实施规范

在进行代码修改前，先输出以下内容：

1. 当前实现概览
2. 问题点或目标
3. 计划修改的文件
4. 风险点
5. 验证方案

在修改完成后，输出以下内容：

1. 实际修改文件
2. 核心改动说明
3. 数据库影响
4. Redis 影响
5. 接口影响
6. 测试与验证结果
7. 剩余风险

---

## 15. 明确禁止事项

- 不要未经说明擅自升级依赖版本。
- 不要未经说明修改公共接口返回结构。
- 不要未经说明删除字段、删除表、删除缓存 key 规则。
- 不要顺手做大规模格式化，除非用户明确要求。
- 不要把业务逻辑塞进 Controller、Mapper、配置类。
- 不要引入与现有架构风格冲突的新框架。
- 不要生成“看起来很多但实际没价值”的样板代码。

---

## 16. Claude 输出要求

Claude 在这个项目里工作时，回答应尽量遵循以下格式：

### 方案类问题
- 先说明当前判断
- 再给出建议方案
- 最后说明风险和取舍

- 先说明改动范围
- 再给出实现方式
- 最后给出验证方式

- 先列最可能原因
- 再给出排查顺序
- 最后给出修复建议

1. 严重问题
2. 一般问题
3. 优化建议
4. 是否建议合并

## 17. 项目优先级原则

当多个目标冲突时，优先级从高到低如下：

1. 正确性
2. 安全性
3. 数据一致性
4. 可维护性
5. 性能
6. 开发速度

---

## 18. 默认偏好

- 默认采用最小可行修改
- 默认保持向后兼容
- 默认不做破坏性重构
- 默认补充必要测试
- 默认评估数据库和缓存影响
- 默认遵循现有代码风格和目录结构
- 优先最小改动，不做无关重构
- Controller 保持轻量，业务逻辑放 Service
- 使用构造器注入，禁止字段注入
- 不直接暴露 Entity 给接口层
- MyBatis-Plus 查询条件必须明确、安全、可读
- 涉及 MySQL 改动时评估索引、锁、数据影响
- 涉及 Redis 改动时评估 key、TTL、一致性和并发风险
- 所有外部输入都视为不可信
- 不在日志中输出敏感信息
- 修改后说明文件、行为、测试和风险
